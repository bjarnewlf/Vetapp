Phasen der Migration
1
Neues Datenmodell
Datenbank + TypeScript-Typen
  • Neuer TypeScript-Typ MedicalEvent mit MedicalEventType: 'vaccination' | 'deworming' | 'checkup' | 'custom'
  • DB-Migration: Neue Tabelle medical_events mit RLS + Datenmigration aus alten Tabellen
  • Schema-Dokumentation aktualisiert
  • notification_id bei Reminders nachgepflegt (F-011)
2
Context-Refactoring
God-Object aufloesen

Der DataContext (~350 Zeilen) wurde in drei spezialisierte Contexts aufgeteilt:

  • PetContext / usePets() — Pets + Documents
  • MedicalContext / useMedical() — MedicalEvents + Reminders
  • VetContactContext / useVetContact() — Tierarzt-Kontakt

Jeder Context mit eigenem State, refresh() und Error-Handling. DataContext vollstaendig geloescht — keine Fassade mehr noetig.

3
Screen-Migration
16+ Screens umgestellt
  • ALLE Screens von useData() auf spezialisierte Hooks umgestellt
  • AddEventScreen: Einheitlich addMedicalEvent() fuer alle Event-Typen
  • PetDetailScreen: Events aus medicalEvents gefiltert
  • HomeScreen: Statistiken auf neues Modell umgestellt
  • AIAssistantScreen + aiService: Neues medicalHistory-Format
  • Edge Function: Auf neues Format angepasst
4
Aufraeumen
Altlasten entfernen
  • Alte Typen Vaccination und Treatment geloescht
  • DataContext.tsx geloescht
  • Rules aktualisiert: supabase.md, state-management.md
  • Zweite Migration: CHECK-Constraint fuer recurrence_interval
Architektur: Vorher vs. Nachher
Vorher
  • 1x DataContext (~350 Zeilen)
  • Tabellen: vaccinations + treatments
  • Hook: useData() in allen Screens
  • CRUD: separate Funktionen pro Typ
  • KI: gemischte Arrays in History
  • Neuer Event-Typ = neue Tabelle
Nachher
  • 3x spezialisierte Contexts
  • Tabelle: medical_events mit type-Feld
  • Hooks: usePets() / useMedical() / useVetContact()
  • CRUD: einheitlich addMedicalEvent()
  • KI: sauberes medicalHistory-Format
  • Neuer Event-Typ = neuer String-Wert

Learnings aus diesem Refactoring
Thema Learning
God-Object-Refactoring Erst Fassade einbauen, dann schrittweise migrieren, dann Fassade loeschen
QA-Timing QA parallel zum Cleanup laufen lassen fuehrt zu False Positives — besser sequentiell
Datenmodell Ein einheitliches Modell ist flexibler als N separate Tabellen — Typen als Wert, nicht als Schema
Custom-Events Custom-Events duerfen nicht nur Reminder sein — sie gehoeren auch in die Gesundheitshistorie
