🔴 Kritisch

1 — Premium-Bypass ohne Kaufverifizierung
Kritisch
src/context/SubscriptionContext.tsx · src/screens/PaywallScreen.tsx

Die Funktion togglePro() schreibt is_premium = true direkt in die Datenbank — ohne jeden Bezahlnachweis. Die PaywallScreen ruft diese Funktion unmittelbar beim Tap auf "Jetzt upgraden" auf, ohne eine echte In-App-Purchase-Transaktion auszulösen oder einen Receipt zu validieren.

// SubscriptionContext.tsx
const togglePro = async () => {
  const newValue = !isPro;
  setIsPro(newValue);
  await supabase.from('profiles')
    .update({ is_premium: newValue }) // ← direkt in DB schreiben
    .eq('id', user.id);
};

// PaywallScreen.tsx
const handleUpgrade = () => {
  togglePro(); // ← kein IAP, kein Receipt
  navigation.goBack();
};

Da die RLS-Policy für profiles dem User erlaubt, sein eigenes Profil zu updaten, kann jeder eingeloggte User sich kostenlos auf Pro setzen — entweder über die UI oder direkt per API-Call.

Empfohlener Fix Echtes In-App-Purchase über expo-iap (oder RevenueCat) integrieren. is_premium darf nur serverseitig (Edge Function) nach erfolgreicher Purchase-Verification gesetzt werden — nie direkt vom Client. Die RLS-Policy für profiles UPDATE sollte das Schreiben von is_premium explizit verbieten.

🟠 Hoch

2 — Rate-Limit Fail-Open in Edge Function
Hoch
supabase/functions/ai-chat/index.ts · Z. 131–133

Wenn die Rate-Limit-Abfrage gegen ai_usage einen DB-Fehler wirft, wird der Request bewusst durchgelassen statt geblockt ("Fail-open").

if (usageError) {
  // Fail-open: Bei DB-Fehler Request durchlassen, nur loggen
  console.error('[ai-chat] Rate-Limit-Abfrage fehlgeschlagen:', usageError.message);
}
// → weiter ohne Limit-Check

Ein Angreifer kann das Rate-Limit systematisch umgehen, wenn er einen Zustand erzeugt, in dem die DB-Abfrage konstant fehlschlägt (z.B. Tabelle nicht vorhanden, Timeout, falsche RLS). Das öffnet die Tür für unkontrollierte Kosten über die Anthropic API.

Empfohlener Fix Auf Fail-closed wechseln: Bei usageError den Request mit HTTP 429 oder 503 ablehnen, nicht durchlassen. Alternativ: In-Memory-Fallback-Counter als zweite Absicherung.
3 — ai_usage-Tabelle fehlt im Schema
Hoch
supabase-schema.sql

Die Edge Function setzt eine Tabelle ai_usage voraus (für Rate-Limiting und Usage-Tracking), die jedoch komplett fehlt in supabase-schema.sql. Ohne Tabellendefinition gibt es auch keine Row Level Security-Policies dafür.

Mögliche Konsequenzen: Die Tabelle existiert in der Produktionsdatenbank ohne RLS → alle User können gegenseitig Rate-Limit-Einträge lesen und manipulieren. Oder sie existiert gar nicht → Rate-Limiting greift nie (Fail-open, siehe Finding 2).

Empfohlener Fix Tabellendefinition und RLS-Policies in supabase-schema.sql ergänzen:

CREATE TABLE public.ai_usage (id uuid DEFAULT gen_random_uuid() PRIMARY KEY, user_id uuid REFERENCES auth.users ON DELETE CASCADE NOT NULL, created_at timestamptz DEFAULT now());

RLS aktivieren + Policy: User kann nur eigene Zeilen sehen/einfügen.

🟡 Mittel

4 — Anon Key als Authorization-Header
Mittel
src/services/aiService.ts · Z. 46–48

Der HTTP-Call an die Edge Function schickt den öffentlichen Supabase Anon Key im Authorization-Header, während der echte User-JWT als Custom-Header x-user-token mitgeliefert wird.

headers: {
  'Authorization': `Bearer ${process.env.EXPO_PUBLIC_SUPABASE_ANON_KEY!}`, // ← Anon Key
  'x-user-token': session.access_token,  // ← User-JWT als Custom-Header
}

Das ist unkonventionell. Standard bei Supabase Edge Functions ist, den User-JWT direkt als Bearer Token zu verwenden. Das aktuelle Pattern macht den Code schwerer verständlich und könnte bei zukünftigen Änderungen zu Fehlkonfigurationen führen.

Empfohlener Fix Authorization: Bearer <session.access_token> verwenden und den x-user-token-Custom-Header entfernen. Die Edge Function dann über req.headers.get('authorization') den JWT holen.
5 — Signed URLs für Dokumente mit 1-Jahr-Gültigkeit
Mittel
src/utils/fileUpload.ts · Z. 84–87

Beim Upload eines Dokuments wird direkt eine Signed URL mit 365-Tage-Gültigkeit generiert und in der Datenbank gespeichert. Zum Vergleich: Pet-Fotos bekommen 1 Stunde.

createSignedUrl(path, 60 * 60 * 24 * 365); // ← 365 Tage

Eine einmal geleakte URL (Clipboard, Screenshot, Log) bleibt ein Jahr lang gültig — ohne Authentifizierung, direkt abrufbar. Gerade bei medizinischen Dokumenten (Befunde, Impfpässe) ist das ein erhöhtes Risiko.

Empfohlener Fix Signed URLs nicht persistieren, sondern on-demand generieren (wie bereits bei Pet-Fotos gemacht). Gültigkeitsdauer auf max. 24–48 Stunden reduzieren.
6 — Storage-Bucket-Policies nicht im Schema definiert
Mittel
supabase-schema.sql

Das Schema definiert RLS für alle Datenbanktabellen, aber keine Bucket-Policies für den pet-documents-Storage-Bucket. Wenn dieser Bucket in der Supabase-Konsole auf "Public" gesetzt ist, sind alle hochgeladenen Dokumente und Fotos öffentlich per direkter URL zugänglich — unabhängig von der Datenbankzugriffskontrolle.

Empfohlener Fix Bucket als privat konfigurieren und Storage-Policies im Schema dokumentieren (z.B. als Kommentar-Block mit den SQL-Statements). Bucket-Policy: User darf nur Dateien unter seinem eigenen userId/-Pfad lesen/schreiben.

🔵 Niedrig

7 — Keine E-Mail-Validierung clientseitig
Niedrig
src/screens/RegisterScreen.tsx · Z. 20–27

Die Registrierung prüft nur ob die Felder nicht leer sind und das Passwort ≥ 6 Zeichen hat. Ein Regex-Check auf das E-Mail-Format fehlt — Supabase fängt das zwar serverseitig ab, aber der Fehler kommt erst nach dem API-Call zurück.

Empfohlener Fix Einfachen E-Mail-Check vor dem API-Call ergänzen: /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email)
8 — Auth-Aktivitäten in Production-Logs
Niedrig
src/services/aiService.ts · Z. 29, 31

Token-Refresh-Aktivitäten werden geloggt, was in Production-Crash-Logs (z.B. Sentry) sensitiven Kontext erzeugen kann.

console.log('[aiService] Token refreshed erfolgreich');
console.warn('[aiService] Refresh fehlgeschlagen:', refreshError?.message);
Empfohlener Fix Logs in Production-Builds entfernen oder hinter ein __DEV__-Flag setzen.

Offene Fragen / Entscheidungsbedarf

  • IAP-Integration: Welcher Anbieter? expo-iap (nativ) oder RevenueCat (managed)? RevenueCat wäre einfacher zu implementieren und zu warten.
  • Soll is_premium in der RLS-Policy explizit gegen Client-Writes gesperrt werden, oder reicht es, den togglePro-Flow zu ersetzen?
  • Signed URLs für Dokumente: Werden diese URLs irgendwo außerhalb des App-States persistiert (Push Notifications, E-Mails)? Das beeinflusst die Strategie.
  • Ist der pet-documents-Bucket aktuell public oder private? Bitte in der Supabase-Konsole prüfen.
  • Priorisierung: Sollen alle 8 Findings vor dem nächsten Release angegangen werden oder nur die kritischen/hohen?
